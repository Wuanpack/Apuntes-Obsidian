ISO/IEC 19249:2017 *Information technology - Security techniques - Catalogue of architectural and design principles for secure products, systems and applications*.

La ISO/IEC 19249 lista cinco principios de arquitectura:

1. Separación de Dominio: Todos los conjuntos de componentes relacionados es agrupado como una sola entidad; los componentes pueden ser aplicaciones, data, u otros recursos. Cada entidad va a tener su propio dominio y serpa asignado a un conjunto común de atributos de seguridad. Por ejemplo, consideremos los niveles de privilegio del procesador x86: el núcleo del sistema operativo puede ejecutarse en el anillo 0 (el nivel de mayor privilegio). En cambio, las aplicaciones en modo usuario pueden ejecutarse en el anillo 3 (el nivel de menor privilegio). La separación de dominios está incluida en el modelo de Goguen-Meseguer.
2. Layering: Cuando un sistema está estructurado en tantos niveles o capas de abstacción, se vuelve posible imponer políticas de seguridad en distintos niveles; además, sería factible validar la operación. Consideremos el modelo OSI con sus 7 capas de red. Cada capa del modelo OSI proporciona servicios específicos a la capa superior. Este layering hace posible imponer políticas de seguridad y validar fácilmente que el sistema está trabajando como debe.
3. Encapsulation: En programación orientada a objetos, escondemos las implementaciones de bajo nivel y prevenimos la manipulación directa de los datos en un objeto al proporcionar métodos específicos para ese propósito.
4. Redundancia: Este principio asegura integridad y disponibilidad. Hay muchos ejemplos relacionados a la redundancia. Considera el caso del servidor de un hardware que tiene dos fuentes de poder; si una fuente falla, el sistema puede continuar con sus funciones. Considere una configuración RAID 5 con tres discos: si uno falla, los datos se mantienen disponibles usando los 2 discos restantes. Además, si los datos son cambiados inapropiadamente en uno de los discos, podría ser detectado por paridades, asegurando la integridad de los datos.
5. Virtualization: Con los avances de los servicios cloud, la virtualización se ha hecho más común y popular. El concepto de virtualización es compartir un solo conjunto de hardware entre múltiples sistemas operativos. La virtualización proporciona capacidades de sandbox que mejoran los límites de seguridad, detonación segura, y observación de programas maliciosos.

Además, el ISO/IEC 19429 enseña 5 principios de diseño:

1. Least Privilege: Este principio enseña que deberías proporcionar la menor cantidad de permisos para alguien para realizar sus tareas y nada más. Por ejemplo, si un usuario necesita ser capaz de ver un documento, deberías darle permisos de lectura y no de escritura.
2. Attack Surface Minimisation: Todos los sistemas tienen vulnerabilidades que un atacante podría usar para comprometer un sistema. Algunas vulnerabilidades son conocidas, mientras que otras aún tienen que ser descubiertas. Estas vulnerabilidades representan riesgos que deberíamos apuntar a minimizar. Por ejemplo, en uno de los pasos para mejorar la seguridad de un sistema Linux, podríamos deshabilitar cualquier servicio no necesitado.
3. Centralized Parameter Validation: Muchas amenazas son debido a que el sistema está recibiendo input, especialmente de usuarios. Inputs inválidos pueden ser usados para explotar vulnerabilidades en el sistema, como DoS o ejecución de código remoto. Por lo tanto, la validación de parámetros es un paso necesario para garantizar un estado del sistema correcto.
4. Centralized General Security Services: Como un principio de seguridad, deberíamos apuntar a centralizar todos los servicios de seguridad. Por ejemplo, podríamos crear un servidor centralizado para autenticación. Por supuesto, podrías tomar medidas apropiadas para garantizar la disponibilidad y prevenir crear un solo punto de falla.
5. Preparing for Error and Exception Handling: Cuando construimos un sistema, deberíamos tomar en cuenta que los errores y excepciones ocurren y van a ocurrir. Este principio enseña a que el sistema debería ser diseñado a fallar de forma segura; por ejemplo, si un firewall crashea, debería bloquear todo el tráfico en vez de permitir todo el tráfico. Además, deberíamos tener cuidado que los mensajes de error no filtren información que podríamos considerar confidencial.
